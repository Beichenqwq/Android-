Activity

Activity是Android app中的一个重要的组件。它为用户提供可交互的一个窗口，通常该窗口充满一屏，但有时比一屏小，并浮在其他窗口上。

一个app通常由多个Activity组成，这些Activity松散的联系在一起。在这些Activity中的某一个被用于启动app，该Activity称为main activity。每个Activity还可以启动其他的Activity，甚至是其他的App，每次新的Activity被启动的时候，前一个Activity会停止，但是该停止的Activity会被保留在一个称为back栈的栈中。当一个新的Activity启动的时候，该Activity会被推入back栈的栈顶，并呈现给用户。Back栈也遵守栈的一般规则，既是LIFO，因此，当用户结束当前的Activity并按下"返回"按钮的时候，该Activity就会从栈顶弹出（并销毁），前一个Activity成为栈顶并被再次激活。

Activity具有生命周期，每当Activity进入某个生命周期，相应的回调方法会被调用，在该回调方法中可以做一些特定的工作。

1. 创建一个Activity的流程

为了创建Activity，你必须创建一个Activity的子类（或其子类的子类）。在你的子类中，你需要实现Activity生命周期的某些回调方法，最重要的一个回调方法如下：
onCreate()

你必须实现该方法，系统在创建你的Activity的时候会调用它。在该方法的实现中，可以为Activity初始化必要的组件。其中最重要的是，在这里调用setContentView()方法来完成Activity的布局。

1.1. 设计并实现用户接口

Activity 的用户界面是由层级式视图 — 衍生自 View 类的对象 — 提供的。每个View控制 Activity 窗口内的特定矩形空间，可对用户交互作出响应。 例如，View可以是在用户触摸时启动某项操作的按钮。

可以利用 Android 提供的许多现成视图设计和组织布局。"小部件"是提供按钮、文本字段、复选框或仅仅是一幅图像等屏幕视觉（交互式）元素的视图。 "布局"是衍生自 ViewGroup 的视图，为其子视图提供布局模型，例如线性布局、网格布局或相对布局。 还可以为 View 类和 ViewGroup 类创建子类（或使用其现有子类）来自定义小部件和布局，然后将它们应用于Activity布局。

利用视图定义布局的最常见方法是借助保存在应用资源内的 XML 布局文件。这样一来，可以将用户界面的设计与定义同 Activity 行为的源代码分开维护。 可以通过 setContentView() 将布局设置为 Activity 的 UI，从而传递布局的资源 ID。不过，也可以在 Activity 代码中创建新 View，并通过将新 View 插入 ViewGroup 来创建视图层次，然后通过将根 ViewGroup 传递到 setContentView() 来使用该布局。

1.2. 配置manifest

为了在系统中存取你的activity，你必须在manifest中声明它。

1.2.1. 在manifest中声明activity

打开你的manifest文件，在<application>中添加一个<activity>元素，例如：

<manifest ... >
  <application ... >
      <activity android:name=".ExampleActivity" />
      ...
  </application ... >
  ...
</manifest >

这里还可以定义其他属性，比如label、icon等，而android:name属性是唯一必须有的属性，这里指定了该activity的类名（别忘了类名前那个点）。需要注意的是，一旦你发布了你的app，你就不能改变这个值了，因为一旦改变了，你就可能破坏了一些诸如app快捷方式等功能。

1.2.2. 配置intent过滤器

在<activity>中使用<intent-filter>可以指定各种intent过滤器，过滤器的作用就是声明其他app组件是如何激活该activity的。本质上讲，Intent过滤器表明了该Activity可以做什么，从而Android系统通过提交给它的Intent 和 Intent过滤器来进行匹配，从而决定哪个Activity可以被运行。

当你使用Android Studio 创建一个新的app的时候，会自动创建一个根activity，它里面会包含一个intent过滤器，该过滤器声明本activity对"main"动作进行响应，并处于"launcher"这个类别中，如下：

<activity android:name=".ExampleActivity" android:icon="@drawable/app_icon">
    <intent-filter>
        <action android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>

上面的<action>元素指定该activity是app的主入口点，<category>元素指定该activity放在系统的"应用"中，如下图中的拨号等。

如果你的app中的activity不允许在你的app之外被启动，那么就不需要其他的intent过滤器了。在你的app中只有一个activity 中有 main 动作和 launcher 类别。

如果你的app中的activity也可以被其他的app所启动，这时还必须定义其他的intent过滤器。

1.2.3. 声明权限

activity不但可以在你的app中被启动，甚至可以在其他app中来调用你的app的某个activity。在<activity>标签中，你可以使用android:permission属性来定义使用本Activity的权限名称，而其他的app若需要启动本activity，也需要先使用<uses-permission>来声明此权限，从而获得启动该Activity的权限。。

比如：如果你的app需要使用某个称为SocialApp的app来在社交媒体上分享帖子，首先SocialApp这个app需要定义调用它的权限，如下：

<manifest>
<activity android:name="...."
   android:permission="com.google.socialapp.permission.SHARE_POST"

/>

你的app如果要使用SocialApp这个app，需要在manifest中声明权限，如下：

<manifest>
   <uses-permission android:name="com.google.socialapp.permission.SHARE_POST" />
</manifest>

1.3. 启动 Activity

通过调用 startActivity()，并传递给其相应的 Intent 来启动一个 Activity。通过该Intent 对象，可以指定启动具体的某个 Activity 或描述需要执行的操作类型（系统会选择合适的 Activity，甚至是来自其他应用的 Activity）。 Intent 对象还可能携带一些数据，传递给所启动的 Activity 。

显式Intent

在自定义应用内工作时，可以直接指定启动某个已知 Activity。 通过使用该Activity类名创建一个显式Intent来启动该 Activity 。 例如，可以通过以下代码让一个 Activity 启动另一个名为 SignInActivity 的 Activity：

Intent intent = new Intent(this, SignInActivity.class);
startActivity(intent);

隐式Intent
如果自定义应用需要执行某项操作，例如发送电子邮件、短信或状态更新，而该应用自身可能不具有执行此类操作所需的 Activity，可以改为利用设备上其他应用提供的 Activity 为您执行这些操作。 这便是 Intent 对象的真正价值所在 — 可以创建一个 Intent 对象，对需要执行的操作进行描述，系统会从其他应用启动相应的 Activity。 如果有多个 Activity 可以处理该 Intent，则用户可以选择要使用哪一个。 例如，如果用户需发送电子邮件，可以创建以下 Intent：

Intent intent = new Intent(Intent.ACTION_SEND);
intent.putExtra(Intent.EXTRA_EMAIL, recipientArray);
startActivity(intent);

putExtra的参数是一个键值对，第一个参数为key，第二个为value，这里，添加到 Intent 中的 EXTRA_EMAIL 是一个字符串数组，其中包含应将电子邮件发送到的电子邮件地址。 当电子邮件应用响应此 Intent 时，它会读取 extra 中提供的字符串数组，并将它们放入电子邮件撰写窗体的"收件人"字段。 在这种情况下，电子邮件应用的 Activity 启动，并且当用户完成操作时，会返回到您的 Activity ，并继续执行。

启动 Activity 以获得结果

某些情况下，可能需要从被启动的 Activity 获得返回结果。在这种情况下，可以通过调用 startActivityForResult()（而非 startActivity()）来启动 Activity。 要想在随后收到后续 Activity 的结果，需要实现 onActivityResult() 回调方法。 当后续 Activity 完成时，它会再使用 Intent 向 onActivityResult() 方法返回结果。

例如，用户选取联系人，对该联系人中的信息执行某项操作。 可以通过以下代码创建此类 Intent 并处理结果：

private void pickContact() {
    // Create an intent to "pick" a contact, as defined by the content provider URI
    Intent intent = new Intent(Intent.ACTION_PICK, Contacts.CONTENT_URI);
    startActivityForResult(intent, PICK_CONTACT_REQUEST);
}

@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
    // If the request went well (OK) and the request was PICK_CONTACT_REQUEST
    if (resultCode == Activity.RESULT_OK && requestCode == PICK_CONTACT_REQUEST) {
        // Perform a query to the contact's content provider for the contact's name
        Cursor cursor = getContentResolver().query(data.getData(),
        new String[] {Contacts.DISPLAY_NAME}, null, null, null);
        if (cursor.moveToFirst()) { // True if the cursor is not empty
            int columnIndex = cursor.getColumnIndex(Contacts.DISPLAY_NAME);
            String name = cursor.getString(columnIndex);
            // Do something with the selected contact's name...
        }
    }
}

上例显示的是，在处理 Activity 结果时应该在 onActivityResult() 方法中使用的基本逻辑。 第一个条件检查请求是否成功（如果成功，则resultCode 将为 RESULT_OK）以及此结果响应的请求是否已知 — 在此情况下，requestCode与随 startActivityForResult() 发送的第二个参数匹配。 代码通过查询 Intent 中返回的数据（data 参数）从该处开始处理 Activity 结果。

1.4. 关闭一个Activity

可以使用finish()方法或者finishActivity (int requestCode)来关闭一个activity，但是我们不建议这么做，因为activity有其生命周期，系统会对activity的生命周期进行管理。

2.2. 构建一个Intent

Android应用程序中各个组件之间必须通过一定的手段把它们联系在一起，它们方可协调运作。这个联系手段就是Intent。Intent 表示一个应用"执行某项操作的意向"。 这里我们通过使用Intent可以启动另一个Activity。

在 MainActivity.java 中，添加 EXTRA_MESSAGE 常量和 sendMessage() 代码，这里的常量和方法名不见得非得这么定义，我们这里是示例，只是这么定义了而已，如此处所示：

public class MainActivity extends AppCompatActivity {
    public static final String EXTRA_MESSAGE = "com.example.myfirstapp.MESSAGE";
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
    }

    /** 当用户点击send按钮的时候调用本方法 */
    public void sendMessage(View view) {
        //创建一个Intent对象
        Intent intent = new Intent(this, DisplayMessageActivity.class);
        //通过获取布局中的edittext的id来操作该edittext
        EditText editText = (EditText) findViewById(R.id.editText);
        //获取edittext中输入的文本
        String message = editText.getText().toString();
        //将这个文本字符串以键值对的形式附加到Intent中
        intent.putExtra(EXTRA_MESSAGE, message);
        //通过intent启动另一个activity，同时传递数据
        startActivity(intent);
    }
}

Android Studio 可能将再次遇到 Cannot resolve symbol 错误，因此，请按 Alt + Enter（在 Mac 上，则按 Option + Return），导入相应的包。

注意：此时对DisplayMessageActivity的调用仍然会显示出错，我们这时可以先忽略错误。

在上面的代码中，

  1. Intent构造方法用到了两个参数：
    • Context作为第一个参数（这里是this，是本方法所在的Activity，而Activity是Context的子类）
    • 第二个参数为系统发送Intent的目标，为某应用组件的Class。也就是被启动的组件的类名。
  2. 使用findViewById()来获取EditText元素
  3. 将文本内容赋值给message变量，使用putExtra()方法将message加入intent。Intent可以携带称为extra的键值对数据，putExtra()方法的第一个参数为键，第二个参数为值。
  4. 为了使下一个Activity能够检索到extra数据，要定义一个public常量作为intent的extra数据的键。通常我们用应用的包名作为前缀，以保证键的唯一性。
  5. 最后，将Intent对象作为参数，调用startActivity()方法，系统接收到调用之后，启动Intent中指定的Activity。
    有时候，你需要被启动的activity要反馈回结果，此时，被启动的activity是通过startActivityForResult()来启动。获取反馈结果是通过实现onActivityResult()回调方法来实现的。被启动的activity在执行完毕以后会将结果放在一个Intent中，返回给onActivityResult()方法。

2.5. 获取 Activity 的返回结果

前面的启动方式，无法获取被启动activity的反馈结果。若需要获取反馈结果，需要调用 startActivityForResult()（而不是 startActivity()）来启动另一个activity。

例如，您的应用可启动相机应用并接收拍摄的照片作为结果。或者，您可以启动"联系人"应用以便用户选择联系人，并且将获取联系人详细信息作为结果。

当然，响应的 Activity 必须设计为可以返回结果。在该被调Activity中生成 Intent 对象，将反馈信息附加在该Intent上，并将该Intent提交给系统，从而发送反馈结果。主调 Activity 通过 onActivityResult() 回调方法来接收反馈信息。

注：当您调用 startActivityForResult() 时，可以使用显式或隐式 Intent。当启动的是自己App中的 Activity 以接收结果时，应使用显式 Intent 确保可收到预期结果。

对于startActivityForResult() 方法中的intent与之前介绍的并没有什么差异，只不过是需要在这个方法里面多添加一个int类型的参数。

这个integer的参数叫做"request code"，用于标识你的请求。当你接收到result Intent时，可以从回调方法里面的参数去判断这个result是否是你发出的请求所发回的结果。

例如，下面是一个启动activity来选择联系人的例子：

static final int PICK_CONTACT_REQUEST = 1;  // The request code
...
private void pickContact() {
    Intent pickContactIntent = new Intent(Intent.ACTION_PICK, Uri.parse("content://contacts"));
    //pickContactIntent.setType(Phone.CONTENT_TYPE); // Show user only contacts w/ phone numbers
    startActivityForResult(pickContactIntent, PICK_CONTACT_REQUEST);
}

当被启动的activity完成并返回结果的时候，系统会调用主调activity中的onActivityResult()这个回调方法。这个方法包括三个参数：

  • 你传递给startActivityForResult()的request code
  • 第二个activity指定的result code。如果操作成功通常是RESULT_OK ，如果用户没有操作成功，而是直接点击回退或者其他什么原因，那么则是RESULT_CANCELED
  • 第三个参数则是intent,它包含了返回的result数据部分

下面是如何处理"pick a contact"意图的结果的例子：

@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
    // Check which request we're responding to
    if (requestCode == PICK_CONTACT_REQUEST) {
        // Make sure the request was successful
        if (resultCode == RESULT_OK) {
            // The user picked a contact.
            // The Intent's data Uri identifies which contact was selected.

            // Do something with the contact here (bigger example below)
        }
    }
}

所谓Activity的生命周期是指Activity从被创建到销毁的过程。每个activity在它的生命周期里会在不同的状态之间转换。在这个转换过程中，android系统会调用一系列的生命周期方法，在这些方法中，你可以完成相应功能。

3.1. Activity生命周期的状态及转换

Activity 基本上以三种稳定状态存在：

  • 继续(Resumed)
    此 Activity 位于屏幕前台并具有用户焦点，用户可以和它进行交互。（有时也将此状态称作"running"。）

  • 暂停(Paused)
    该activity被其他activity部分遮盖，这时该activity不会接收用户的输入也不会执行任何代码。 暂停的 Activity 处于完全活动状态（Activity 对象保留在内存中，它保留了所有状态和成员信息，并与窗口管理器保持连接），但在内存极度不足的情况下，可能会被系统终止。

  • 停止(Stopped)
    该 Activity 完全隐藏，处于后台，这时activity实例和它的所有信息都保持着，但不能执行任何代码。 已停止的 Activity 同样仍处于活动状态（Activity 对象保留在内存中，它保留了所有状态和成员信息，但未与窗口管理器连接）。 不过，它对用户不再可见，在内存短缺时可能会被系统终止。

如果 Activity 处于暂停或停止状态，系统可通过要求其结束（调用其 finish() 方法）或直接终止其进程，将其从内存中删除。（将其结束或终止后）再次打开 Activity 时，必须重建。

与Activity的生命周期联系在一起的有六个生命周期方法：onCreate(), onStart(), onResume(), onPause(), onStop(), 和 onDestroy()。Android系统在当Activity在不同生命周期之间切换时，调用相应回调方法来执行程序功能。

由于你的activity的复杂性，你可能不必实现所有的生命周期方法，但你应该了解这些方法。实现这些生命周期方法会使得你的app表现良好。

在activity的生命周期状态的切换中，系统会按照一个序列调用相应的生命周期方法。如在下图中，当系统创建一个新的activity的实例的时候，每次回调方法把activity的状态推向顶端。在顶端，用户可以同该activity进行交互。
当用户开始离开activity的时候，系统调用其他的方法从顶端向下移动。在某些情况下，activity还能从某些点回到顶端。

上图中只有三种状态是稳定的，activity可以稳定的处于这三种状态：Resumed、Paused及Stopped。除了这三种状态，其他状态只是过渡状态，会迅速切换到另外的状态，既是当系统调用onCreate()以后会立即调用onStart()，再紧跟着调用onResume()方法。

  • Activity 的整个生命周期发生在 onCreate() 调用与 onDestroy() 调用之间。Activity 应在 onCreate() 中执行"全局"状态设置（例如定义布局），并在 onDestroy() 中释放所有其余资源。例如，如果您的 Activity 有一个在后台运行的线程，用于从网络上下载数据，它可能会在 onCreate() 中创建该线程，然后在 onDestroy() 中停止该线程。

  • Activity 的可见生命周期发生在 onStart() 调用与 onStop() 调用之间。在这段时间，用户可以在屏幕上看到 Activity 并与其交互。例如，可以在 onStart() 中注册一个 BroadcastReceiver 以监控影响 UI 的变化，并在用户无法再看到显示的内容时在 onStop() 中将其取消注册。在 Activity 的整个生命周期，当 Activity 在对用户可见和隐藏两种状态中交替变化时，系统可能会多次调用 onStart() 和 onStop()。

  • Activity 的前台生命周期发生在 onResume()调用与 onPause() 调用之间。在这段时间，Activity 位于屏幕上的所有其他 Activity 之上，并具有用户输入焦点。 Activity 可频繁转入和转出前台 — 例如，当设备转入休眠状态或出现对话框时，系统会调用 onPause()。 由于此状态可能经常发生转变，因此这两个方法中应采用适度轻量级的代码，以避免因转变速度慢而让用户等待。

3.2. 实现生命周期回调方法

当一个 Activity 转入和转出上述不同状态时，系统会调用相应的回调方法，我们可以override这些回调方法来执行一些状态切换时需要执行的操作。以下框架 Activity 包括每一个基本生命周期方法：

public class ExampleActivity extends Activity {
    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // The activity is being created.
    }
    @Override
    protected void onStart() {
        super.onStart();
        // The activity is about to become visible.
    }
    @Override
    protected void onResume() {
        super.onResume();
        // The activity has become visible (it is now "resumed").
    }
    @Override
    protected void onPause() {
        super.onPause();
        // Another activity is taking focus (this activity is about to be "paused").
    }
    @Override
    protected void onStop() {
        super.onStop();
        // The activity is no longer visible (it is now "stopped")
    }
    @Override
    protected void onDestroy() {
        super.onDestroy();
        // The activity is about to be destroyed.
    }
}

注：正如以上示例所示，在实现这些生命周期方法时必须始终先调用超类实现，然后再执行任何其它操作。

3.2.1. Activity的整体生命周期：onCreate()与onDestroy()

  1. 调用onCreate()方法启动Activity

当用户从home屏幕选择启动你的app的时候，系统调用你指定的activity的onCreate()方法，这个activity是你的app的主要执行入口。

可以通过AndroidManifest.xml文件指定你的主activity。你的app的主activity在manifest中必须使用<intent-filter>进行声明，<intent-filter>中必须包含MAIN这个action和LAUNCHER这个category，MAIN和LAUNCHER这两个，缺一不可，只要有一个没有定义，你的app图标就不会显示在你的应用程序表中。

大部分的app都可能包含若干不同的activity来执行不同的action。无论是什么样的activity，系统都是通过调用onCreate()方法来创建新的Activity实例。

你必须实现onCreate()方法来实现基本的应用程序启动逻辑，在Activity的整个生命周期中只执行一次这个方法。例如：在onCreate()方法中应该定义UI及可能初始化一些成员变量。

例如：下例中的onCreate()方法执行一些基本的Activity的设置，诸如声明UI（在XML布局文件中定义），定义成员变量，以及配置UI。

TextView mTextView; // 用于引用布局中的 text view 成员变量

@Override
public void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);

    // 设置本Activity的UI布局
    // 该布局定义为 res/layout/main_activity.xml
    setContentView(R.layout.main_activity);

    // 初始化成员变量TextView，以后我们可以操作它
    mTextView = (TextView) findViewById(R.id.text_message);

}

一旦onCreate()执行结束，系统随后立刻调用onStart()和onResume()方法，你的Activity不会停留在Created和Started状态。技术上，当系统调用onStart()以后，Activity就成为可见的，但是onResume()随后立刻执行，然后Activity处于Resumed状态，直到它的状态因为一些原因而发生改变，诸如当接到电话、用户转向其它Activity、或者屏幕被关掉。如下图：

大部分的app不需要实现该方法，因为已经在onPause()和onStop()中执行了大部分的清理工作。 但是如果你的Activity的onCreate()中包含一些后台线程或者其它长期占有的资源，这个方法是防止内存泄露的最后一个机会。如没有适当的关闭会引起潜在的内存泄露，你应该在onDestroy()中kill掉它们。

@Override
public void onDestroy() {

    ...
}

3.2.2. Activity的前台生命周期：onPause()与onResume()

在正常的app使用过程中，前台的activity有时候会被其它可见的组件遮住，从而引起activity暂停。例如，当一个半透明的activity打开时（如对话框），被覆盖的那个activity就会暂停。只要是该activity是部分可见的，但不是焦点activity，它就保持暂停。

一旦activity完全被遮盖或者不可见，它就处于停止状态了。

当你的activity进入暂停状态的时候，系统会调用该activity的onPause()方法，停止正在执行的操作，比如视频的播放。

如果用户返回到处于暂停状态的activity时，系统会调用onResume()方法，该Activity会回到Resumed状态。

注意：当你的activity收到了对onPause()的调用，意味着可能还会回来，但也可能就此离开了你的app。

  1. 暂停activity

当系统调用activity的onPause()方法的时候，意味着你的activity依然部分可见，但是大部分情况是意味着用户即将离开该activity并进入Stopped状态，在onPause()方法中，你可能会做下面的操作：

  • 停止动画或其他可能消耗CPU的动作
  • 提交未保存的改变，但只有用户确实希望保存的才保存（如email草稿）
  • 释放系统资源，诸如广播接收器，对传感器的操作及任何可能影响电池用量而用户不再需要的资源。

例如，如果你的应用使用了Camera，在onPause()方法中就需要释放它。

@Override
public void onPause() {
    super.onPause();  // 首先始终要调用父类中的onPause方法

    // 例：释放摄像头，因为在暂停状态的时候，我们不需要使用它
    // 其他的activity有可能使用它
    if (mCamera != null) {
        mCamera.release()
        mCamera = null;
    }
}

通常，在这里，应该尽量避免大量执行CPU操作，比如写数据库，因为它会拖慢另一个activity的显示（类似这种大量的任务应该放在onStop()方法中）。在onPause()方法中应该执行尽量简单的操作。

  1. 重回activity

当你的activity从暂停状态重启时，系统会调用onResume()方法。

注意每次你的activity在前台运行的时候，都会调用这个方法，包括第一次调用的时候。在这个方法中，这里你应该初始化那些你在onPause()中释放的组件，以及其它每次进入Resumed状态都应该初始化的（比如开始动画等）
下例是上例在onPause()的反操作，它初始化了camera。

@Override
public void onResume() {
    super.onResume();  // 首先始终要调用父类中的onResume方法

    // 再次获取摄像头
    if (mCamera == null) {
        initializeCamera(); // 初始化摄像头
    }
}

3.2.3. Activity的可见生命周期：onStop()和onRestart()

停止和重启你的activity在activity的生命周期中是一个重要的过程。下面描述了一些activity会停止和重启的关键场景：

  • 用户打开Recent Apps这个窗口切换到另一个app的时候，当前前台的activity会停止，当用户通过Home屏幕或Recent Apps窗口返回的时候，activity重新启动。
  • 用户在你的app中启动一个新的activity，当前的activity会停止。当用户压下Back按钮的时候，前一个activity会重启。
  • 当在使用app的时候，收到了电话。

Activity类提供了两个生命周期方法，onStop()和onRestart()方法。当activity处于stopped状态的时候，UI不再可见。

注意：因为当activity处于stopped状态的时候，系统内存中保持有你的Activity的实例，有可能不需要执行onStop()和onRestart()方法（甚至是onStart()方法，由于大部分的activity都相对简单，activity停止和重启都会比较容易，你只需要使用onPause()方法来暂停正在执行的动作和断掉系统资源）

如下图：

  1. 停止activity

当activity需要切换到停止状态的时候，系统会调用onStop()方法，该activity就不再可见了，并且会释放几乎所有的不再需要的资源。在activity停止以后，系统如果需要回收系统内存的时候，可能会销毁该activity的实例。在某些情况下，系统只是简单的kill掉app进程而不会调用onDestroy()方法，所以应该在onStop()中释放可能泄露内存的资源，这一点很重要。

尽管在onStop()之前会调用onPause()方法，你应该使用onStop()来执行需要更耗费CPU的操作，诸如向数据库写数据。

下例是在onStop()在保存便条的内容：

@Override
protected void onStop() {
    super.onStop();  // 总是首先调用父类中的onStop方法

    // 保存笔记草稿，因为activity要停止了
    // 我们要保证当前输入的内容不会丢失
    ContentValues values = new ContentValues();
    values.put(NotePad.Notes.COLUMN_NAME_NOTE, getCurrentNoteText());
    values.put(NotePad.Notes.COLUMN_NAME_TITLE, getCurrentNoteTitle());

    //下面是内容解析器的调用
    getContentResolver().update(
            mUri,    // The URI for the note to update.
            values,  // The map of column names and new values to apply to them.
            null,    // No SELECT criteria are used.
            null     // No WHERE columns are used.
            );
}

当你的activity停止以后，activity对象保持驻留在内存中，activity在返回（resume）的时候会再次被调用。你无须重新初始化那些在导向Resumed状态的回调方法中创建的组件。系统会跟踪布局中的每个View的状态，所以如果用户在EditText中输入文本的时候，内容会被保持而不需要保存或恢复。

注意：甚至在系统销毁掉你的activity，它也会在Bundle（用于保存键值对）中保持着View对象的状态（诸如在EditText中的文本），从而如果用户回到同样的activity的实例以后也能恢复这些内容。

  1. 开始/重启你的app

当你的activity从stopped状态回到前台运行的时候，系统会调用onRestart()及onStart()方法，每次你的activity变得可见的时候都会调用onStart()方法（无论是重启还是第一次创建）。但是，只有在activity从stopped状态中恢复的时候才调用onRestart()方法，所以你可以使用该方法来执行那些需要特别恢复的工作。

把onRestart()用于恢复activity的状态不常见。在onStop()方法中通常清理了所有的activity资源，你需要在activity重启的时候重新实例化它们。另外，在activity被第一次创建的时候也需要实例化这些资源。基于这种原因，应该使用onStart()作为onStop()的对应方法。

例如，由于用户很久以后才再次回到你的app中，这时onStart()方法可以校验哪些系统特征是满足的。

@Override
protected void onStart() {
    super.onStart();  // 总是首先调用父类中的onStart方法

    // 本activity既可能重启，也可能首次启动
    // 因此，在这里保证GPS可用
    LocationManager locationManager =
            (LocationManager) getSystemService(Context.LOCATION_SERVICE);
    boolean gpsEnabled = locationManager.isProviderEnabled(LocationManager.GPS_PROVIDER);

    if (!gpsEnabled) {
        // Create a dialog here that requests the user to enable GPS, and use an intent
        // with the android.provider.Settings.ACTION_LOCATION_SOURCE_SETTINGS action
        // to take the user to the Settings screen to enable GPS when they click "OK"
    }
}

@Override
protected void onRestart() {
    super.onRestart();  // 总是首先调用父类中的onRestart方法

    // Activity being restarted from stopped state    
}

名为"执行完毕后能否终止"的列，表示系统是否能在不执行下一行 Activity 代码的情况下，在方法返回后随时终止包含该Activity 的进程。 有三个方法带有"是"标记：(onPause()、onStop() 和 onDestroy())。由于 onPause() 是这三个方法中的第一个，因此 Activity 创建后，onPause() 必定成为最后调用的方法，然后才能终止进程 — 如果系统在紧急情况下必须恢复内存，则可能不会调用 onStop() 和 onDestroy()。因此，应该使用 onPause() 向存储设备写入至关重要的持久性数据（例如用户编辑）。不过，应该对 onPause() 调用期间必须保留的信息有所选择，因为该方法中的任何阻止过程都会妨碍向下一个 Activity 的转变并拖慢用户体验。

在"执行完毕后能否终止"列中标记为"否"的方法可从系统调用它们的一刻起防止包含该 Activity 的进程被终止。 因此，在从 onPause() 返回的时到 onResume() 被调用时，系统可以终止 Activity。在 onPause() 被再次调用并返回前，将无法再次终止 Activity。

注：上表中的定义属于技术上无法"终止"的 Activity 仍可能被系统终止 — 但这种情况只有在无任何其他资源的极端情况下才会发生。

  4. singleTask模式启动的activity在同一个task栈中可以有几个相同的activity的实例？该实例能在栈的哪个位置？
  5. singleInstance模式启动的activity所在的task栈中可以有几个相同的activity的实例？在同一个task栈中还可以有其他activity的实例吗？

launchMode在多个Activity跳转的过程中扮演着重要的角色，它可以决定是否生成新的Activity实例，是否重用已存在的Activity实例，是否和其他Activity实例公用一个task。一个task就是一个用户用于完成某项工作的activity集合。这些activity以栈（back stack）来进行管理，栈底到栈顶的顺序为每个activity的打开顺序。启动一个应用，也就创建一个与之对应的task。

Activity一共有以下四种launchMode：

  • standard
  • singleTop
  • singleTask
  • singleInstance
    我们可以在AndroidManifest.xml配置<activity>的android:launchMode属性为以上四种之一即可。

这就是singleTask模式，如果发现有对应的Activity实例，则使此Activity实例之上的其他Activity实例统统出栈，使此Activity实例成为栈顶对象，显示到屏幕前。
